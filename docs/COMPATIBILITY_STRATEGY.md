# CloudCraft 兼容策略

本文记录 CloudCraft 和配方查看器、其他科技 mod 的长期兼容边界。它是工程约束，不代表当前版本已经接入第三方 API。

## 当前边界

- `common` 只保存平台无关规则、注册定义、数据生成和共享资源。
- `fabric` 与 `neoforge` 只做各自加载器需要的注册、事件、网络和客户端挂接。
- 当前版本不把 JEI、REI、EMI、FE、RF、EU、Fabric Transfer API 或 NeoForge Energy API 接成编译依赖。
- 临时客户端烟测脚本可以下载 JEI 运行游戏，但这不等于 CloudCraft 运行时依赖 JEI。
- 第三方兼容必须是 optional integration；缺失目标 mod 时 CloudCraft 本体仍应能编译、启动和进入世界。

## 配方查看器

CloudCraft 的配方查看器接入顺序应先展示玩家最容易困惑的非原版流程，再补普通合成链。

1. JEI 优先作为第一个客户端验证目标，因为当前烟测脚本已经能加载 JEI。
2. REI 和 EMI 后续通过同一套 common 展示描述接入，避免三套重复规则。
3. `gas_state_converter` 应有独立展示分类，至少展示：
   - 积云碎片作为催化剂，把附近气态云凝结为同属固态云。
   - 卷云纤维作为催化剂，把附近固态云汽化为同属气态云。
   - 该机器没有 GUI 时，展示应明确为世界交互流程，而不是伪装成有物品槽的机器配方。
4. 压缩空气系统应先展示已有合成链和云内充压说明，不把云方块写成通用燃料。
5. 第一次实现时只允许在平台客户端模块或独立 compat 源集中引用 JEI/REI/EMI API，不允许从 `common` 玩法规则类直接引用。

## 科技 mod 兼容

CloudCraft 的核心能量形态是 compressed air。其他科技 mod 的能量只能辅助压缩、稳压、储压或自动化控制，不能替代云环境与压缩空气循环。

优先级：

1. 数据包和标签兼容优先，例如材料标签、云材料合成替代和机器输入标签。
2. 自动化兼容次之，例如让外部管线可以插入催化剂、读取转换器状态或触发红石。
3. 电力兼容最后做，且只做 optional integration。

电力接入的最低门槛：

- 压缩空气背包、气态转换器、云维度充压环境和配置倍率都已有测试守护。
- 电力只驱动独立的压缩机、稳压器或控制模块，不直接给喷气背包写压力。
- 能量单位换算必须集中在一个 compat 规则类里，并有普通单元测试。
- Fabric 与 NeoForge 的能量 API 不进入 `common` 核心规则包。

## 禁止事项

- 不在 `common` 里直接 import 第三方配方查看器或能量 API。
- 不让 CloudCraft 的基础玩法依赖任一外部科技 mod 才能闭环。
- 不把电力当成压缩空气的通用替代品。
- 不为一个加载器写一套玩法规则，再在另一个加载器复制一份。
