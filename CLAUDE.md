# CloudCraft 项目基础约束

本文只定义长期有效的代码规范与协作约束，不记录阶段计划、当前进度、临时决定，也不提前写死尚未稳定的玩法设想。

## 总体原则
- 先检索，后修改；先复用，后新增
- 优先把规则写清楚，再把实现铺开
- 避免为了“以后可能会用”而提前抽象复杂框架
- 允许小而直接的实现，但不允许重复维护同一条规则

## 多模块边界
仓库采用 `common / fabric / neoforge` 结构，边界必须稳定。

### `common`
只放平台无关内容：
- 核心规则
- 方块、物品、药水、效果、还有其他东西的定义与工厂
- 可复用的纯逻辑
- 共享资源

`common` 不负责持有平台侧运行时注册结果，不依赖平台初始化顺序来回填可变静态字段。

### `fabric / neoforge`
只放平台相关内容：
- 真实注册对象
- 平台事件接入
- 平台 API 适配
- 平台客户端初始化与渲染挂接
- 平台测试接入

平台层应尽量薄，不能把规则真相写在平台模块里。

## 单一真相来源
同一条规则只能有一个真相来源。
- 公共规则应集中在 `common`
- 方块行为、效果行为、药水行为都应调用同一套规则
- 不允许在多个类里复制阈值、扫描范围、目标筛选条件

如果一个改动需要同时修改两处规则判断，通常说明抽象位置不对。

## 命名规则
命名要反映真实语义，不制造误导。
- 名称必须与实际类型一致：`Block`、`Item`、`Effect`、`Potion` 不混用
- 同一概念在注册名、常量名、资源名中尽量共享同一语义词根
- 缩写只在社区通用时使用，否则优先完整语义

如果一个名称需要靠注释解释，通常应该先改名。

## 规则实现规范
- 魔法值必须提取成有语义的常量
- 条件分支优先写成可读规则函数，而不是堆在事件方法里
- 触发入口可以分散，但最终判定应收敛
- 优先把“是否允许”和“实际执行”分开，方便测试与复用

## 复用与重复控制
- 能复用现有工厂、规则类、注册定义时，不新开平行实现
- 不允许为了图快复制一份近似逻辑再微调
- 真有差异时，要把差异点显式提取出来，而不是复制后各自演化

## 资源与数据规范
- 资源名必须与代码中的注册语义一致
- 高重复资源优先考虑 datagen，但前提是不会制造第二套字符串真相来源
- 复杂且低重复的资源可以手写，避免为自动化而自动化
- 语言键、模型、配方、掉落表命名保持可预测

## 测试规范
测试目标是守住规则，不是追求表面覆盖率。
- 纯规则优先写在 `common` 并配单元测试
- 平台行为、真实注册、世界交互优先用平台测试验证
- 每个新增机制至少要有一个最小失败用例能证明规则边界
- 改阈值、改扫描范围、改触发条件时，必须同步更新测试

重点不是“测试多”，而是“关键规则改坏时能立刻报错”。

## 注释与文档规范
- 默认不写解释性注释
- 只在“为什么这样做不直观”时写短注释
- 注释不能重复代码表面含义
- 项目文档记录长期约束，不记录短期过程噪音
- 在项目早期，优先沉淀稳定的工程规范，不提前固化未定型的玩法方向

## 清理要求
每次改动后都应顺手清理：
- 未使用导入
- 调试输出
- 废弃实现
- 模板残留
- 已失效的兼容层

仓库应始终以“当前真实结构”为准，不能保留会误导后续维护者的历史壳子。

## 提交前自检
在提交任何改动前，至少确认：
1. 有没有重复规则实现
2. 有没有破坏 `common / platform` 边界
3. 命名是否仍然准确
4. 调用链是否完整更新
5. 自动验证与必要手工验证是否覆盖到改动点

如果这五点里有任何一点答不清，就先不要继续扩展改动范围。
